## Architecture

### App Layout

The project uses a **domain-module structure** — each Django app is a use-case module nested under a domain package:

```
kardex/                          ← Django project config (settings, root URLs)
GestionDeUsuarios/
  LoginYAutenticacion/           ← CU1: JWT auth (register, login, logout, profile)
  GestionDePersonalDeSalud/      ← CU2: Health staff management
  RegistroYBusquedaDePacientes/  ← CU3: Patient registration & search
  VisualizacionDelExpediente/    ← CU4: Full patient record (T011)
  EdicionDeAntecedentesMedicos/  ← CU5: Medical history editing
AtencionClinica/
  AperturaFichaYColaDeAtencion/  ← CU6: File/queue management
  RegistroDeTriaje/              ← CU7: Triage recording
  ConsultaMedicaSOAP/            ← CU8: SOAP medical consultation
  EmisionDeRecetaMedica/         ← CU9: Prescription issuance
  SolicitudDeEstudios/           ← CU10: Lab/study orders
  FirmaDigitalDeConsulta/        ← CU11: Digital signature on consultations
IA_Blockchain/
  ClasificacionDeUrgenciaConIA/
  PrediccionDeRiesgosClinicos/
  GestionDeIdentidadBlockchain/
  VerificacionDeIntegridadDocumentos/
  ConfiguracionDeConsentimiento/
SeguridadAvanzadaYAdministracion/
  BreakGlass_Solicitud/
  BreakGlass_Aprobacion/
  GestionDePermisosPaciente/
  PanelDeAuditoriaYReportesSNIS/
ml/
  modelo_riesgo.py               ← Logistic Regression: 4 chronic risk predictions
  modelo_triaje.py               ← TF-IDF + Naive Bayes: triage priority NLP
  servicio_ml.py                 ← Singleton service; loaded at Django startup
  modelos_guardados/             ← joblib files (generated by training scripts)
```

### URL Routing

Registered in `kardex/urls.py`:
- `/api/auth/` → `GestionDeUsuarios.LoginYAutenticacion.urls`
- `/api/pacientes/` → `GestionDeUsuarios.VisualizacionDelExpediente.urls`

Each sub-app has its own `urls.py`. When adding a new app's URLs, wire them into `kardex/urls.py`.

### Core Data Model

The central entity chain is:
`Paciente` → `Triaje` (OneToOne to `Consulta`) → `Consulta` (SOAP) → `RecetaMedica` / `OrdenEstudio`

- **Paciente** (`RegistroYBusquedaDePacientes`): identity only — CI + ci_complemento form the business key (unique together). No clinical data here.
- **Antecedente** (`EdicionDeAntecedentesMedicos`): clinical history (allergies, chronic conditions, blood type) — separate from Paciente by design.
- **Triaje** (`RegistroDeTriaje`): vital signs + urgency level. IMC and presion_arterial are computed `@property`, not stored. Contains PostgreSQL CHECK constraints for all vital sign ranges.
- **Consulta** (`ConsultaMedicaSOAP`): SOAP-structured encounter. `codigo_cie10_principal` is required for Bolivian SNIS reporting. State machine: `BORRADOR → COMPLETADA → FIRMADA`. `hash_documento` is SHA-256 anchored to blockchain when signed.

### Authentication

JWT via `djangorestframework-simplejwt`. Access token: 15 min. Refresh token: 30 days with rotation + blacklisting. Default permission is `IsAuthenticated`; public endpoints explicitly set `AllowAny`.

Login response includes `groups` — a list of role names used by clients for role-based UI. Roles: Médico, Enfermera, Administrativo, Laboratorio, Farmacia, Auditor, Director.

### ML Service

`ServicioML` is a Singleton loaded once at startup. **Never import model files directly from views** — always go through the service:

```python
from ml.servicio_ml import ServicioML
svc = ServicioML.obtener_instancia()
svc.clasificar_triaje(texto_sintomas="dolor en el pecho", triaje_id=15)
svc.predecir_riesgo(paciente_id=42, tipo_riesgo="diabetes_tipo2")
```

If joblib files are missing, the service degrades gracefully. Set `ML_MODELS_PATH` env var to change the model directory.

Triage hard overrides (post-NLP): SpO2 < 90% or BP systolic < 80 or > 200 or Glasgow ≤ 8 → ROJO; SpO2 < 94% or EVA ≥ 9 → minimum NARANJA.

### What's Not Yet Implemented

Several apps have empty or stub `views.py`/`urls.py` — they define models but no endpoints. The `ml/servicio_ml.py` calls placeholder ORM queries for `Paciente`, `Antecedente`, `Triaje`, `Consulta` that assume the apps exist (they do now). Celery/Redis, Web3/Blockchain, and MinIO/S3 are declared in `requirements.txt` but not wired up.

### Adding a New App

1. Create the module under the relevant domain package (e.g., `AtencionClinica/NewFeature/`).
2. Add an `apps.py` with `AppConfig` pointing to the correct module path.
3. Register in `INSTALLED_APPS` in `kardex/settings.py`.
4. Wire the app's `urls.py` into `kardex/urls.py`.
5. Run `python manage.py makemigrations && python manage.py migrate`.
